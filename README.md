# Distributed Chat System

A production-grade, horizontally-scalable, real-time chat platform (WhatsApp-class) built as a microservices stack on Node.js, MongoDB, PostgreSQL, Redis, Kafka, S3, Docker, and Kubernetes вҖ” with a Spotify-inspired React frontend.

---

## 1. System Design

### Goals

- **Real-time** 1:1 and group messaging at single-digit-millisecond fanout
- **Durable** message persistence with at-least-once semantics
- **Horizontally scalable** WebSocket layer (no per-server affinity for state)
- **Offline-tolerant**: messages queue, deliver on reconnect, with read receipts
- **Polyglot persistence**: relational for users, document for messages, KV for hot state
- **Production posture**: health probes, graceful shutdown, blue/green deploys, metrics, alerts

### Components

| Service | Purpose | Stack | Default Replicas |
|---|---|---|---|
| `api-gateway` | Single ingress, auth-aware routing, WS upgrade, rate-limit, metrics | Express + http-proxy-middleware | 2 (HPA вҶ’ 10) |
| `auth-service` | Register/login, JWT issue + rotating refresh tokens, user search | Express + Postgres + bcrypt | 2 (HPA вҶ’ 10) |
| `chat-service` | Socket.io WebSockets, presence, typing, sticky sessions | Socket.io + Redis pub/sub + Kafka producer | 3 (HPA вҶ’ 30) |
| `message-service` | REST CRUD for chats, message history, S3 presigned URLs, Kafka consumer вҶ’ MongoDB | Express + Mongoose + KafkaJS + AWS SDK | 3 (HPA вҶ’ 15) |
| `notification-service` | Web Push (VAPID) for offline users, consumes Kafka | Express + web-push + KafkaJS | 2 |
| `frontend` | Spotify-themed React SPA, Zustand store, Vite + Tailwind v4 | React 18 + Socket.io-client | 2 |

### Data flow вҖ” sending a message

```
Client в”Җв”Җmsg:sendв”Җв”Җв–ә chat-service в”Җв”ҖKafka(chat.messages, key=chatId)в”Җв”Җв–ә message-service в”Җв”Җв–ә MongoDB
                          в”Ӯ                                                     в”Ӯ
                          в””в”Җв”ҖRedis pub/sub(chat:messages)в”Җв”Җв–ә other chat-service replicas
                                                                     в”Ӯ
                                                                     в–ј
                                                          io.to(chat:<id>).emit
                                                                     в”Ӯ
                                                                     в–ј
                                                            recipient sockets
                          в”Ӯ
                          в””в”Җв”ҖKafka(chat.notifications)в”Җв”Җв–ә notification-service в”Җв”Җв–ә Web Push (offline only)
```

Key invariants:

1. **Order**: Kafka partitions by `chatId`, so all messages for a chat land on the same partition вҶ’ consumer preserves order.
2. **At-least-once**: Producer waits for `acks=all`. Consumer is idempotent вҖ” Mongo upsert keyed by `messageId` (UUID v4 generated by chat-service).
3. **No state on chat-service**: All cross-instance fanout is via Redis pub/sub. Adding/removing chat-service replicas is zero-coordination.
4. **Stickiness for resume**: Sessions are sticky (Service `sessionAffinity: ClientIP` + Ingress cookie affinity) so reconnects land on the same pod and the in-memory `chat:<id>` Socket.io rooms are preserved.

## 3. Scaling Strategy

### Horizontal scaling вҖ” per service

| Layer | Bottleneck | Scale signal | Strategy |
|---|---|---|---|
| **api-gateway** | CPU on TLS / proxying | CPU 70% | HPA 2вҶ’10. Stateless. |
| **auth-service** | bcrypt CPU + Postgres conns | CPU 70% | HPA 2вҶ’10. PgBouncer in front of Postgres if conn exhaustion. |
| **chat-service** | Open WS file descriptors (~3.5k/pod) and CPU | CPU 60% + custom `socket_io_connected` metric | HPA 3вҶ’30. Sticky sessions preserved on scale-down via `preStop sleep 30` + 60s grace. |
| **message-service** | Kafka consumer throughput + Mongo writes | Kafka lag (custom metric) + CPU | HPA 3вҶ’15. Consumer concurrency = 4 partitions/replica. |
| **notification-service** | Push provider rate limits | Kafka lag | HPA 2вҶ’8. Throttled at provider boundary. |
| **frontend (nginx)** | Almost free | CPU | HPA 2вҶ’6. |

### Data tier scaling

- **Postgres (users)**: Vertical first (CPU + IOPS). Read replicas for `findByEmail`/search if it becomes hot. Connection pool capped at 20/replica Г— вүӨ10 replicas = 200 conns max.
- **MongoDB (messages)**: Sharded by `chatId` once a single replica set saturates. Compound index `{chatId:1, createdAt:-1}` makes cursor-pagination O(log n).
- **Redis**: Single-node initially вҶ’ Redis Cluster + `pub/sub` over `Sharded Pub/Sub` (Redis 7+) when fanout exceeds ~50k msg/s.
- **Kafka**: Partition count вүҘ max chat-service replicas. Increase partitions before scaling consumers past current count. `chat.messages` ships with 12 partitions; increase live by adding partitions (but never reduce вҖ” would break ordering guarantee).
- **S3**: Inherently scalable. Direct browser вҶ’ S3 via presigned URLs offloads bandwidth from the API entirely.

### Geographic scaling (future)

- Active-active regions with Kafka MirrorMaker 2 for cross-region replication of `chat.messages`.
- Region affinity by chat (sticky chat вҶ’ region) to preserve ordering.
- DNS-based routing (Route 53 latency-based / GeoDNS) at the ingress.

## 4. Trade-offs & Decisions

| Decision | Why | Cost |
|---|---|---|
| **Kafka instead of just Redis pub/sub** | Need durable, replayable, ordered message log. Redis pub/sub is fire-and-forget вҖ” a chat-service pod crashing mid-broadcast loses messages. Kafka gives at-least-once with consumer offsets and a DLQ. | More moving parts; Zookeeper (or KRaft) ops cost; ~5вҖ“15ms publish latency. Mitigated by also publishing to Redis pub/sub for *immediate* delivery вҖ” Kafka is the source of truth, Redis is the speed layer. |
| **Sticky WebSocket sessions** | Socket.io rooms are in-memory per pod. Without stickiness, an HTTP polling fallback would fan out to a different pod each request. | Less even load distribution; scale-down requires 60s graceful drain. Acceptable because WS connections are long-lived (minutes to hours). |
| **MongoDB for messages, Postgres for users** | Messages are append-heavy, schema-light, and benefit from horizontal sharding by `chatId`. Users are relational (email uniqueness, refresh tokens, contacts FK) and benefit from ACID + JOINs. | Two database technologies to operate. Worth it: forcing messages into Postgres would require manual sharding. |
| **JWT access (15m) + rotating refresh in Postgres** | Stateless auth at the gateway (no DB hit per request) but with revocability (kill all refresh tokens for a user). Rotation prevents replay if a refresh token leaks. | Slightly more complex than session cookies. Refresh tokens are hashed with SHA-256 before storage so DB compromise doesn't yield usable tokens. |
| **Direct browser вҶ’ S3 uploads via presigned URLs** | Offloads bandwidth from the API entirely. A 50MB video upload doesn't touch any of our pods. | Two round-trips for upload (get URL, then PUT). Worth it at scale вҖ” bandwidth on the API tier is the most expensive resource. |
| **Optimistic UI with `pending` status** | Messages render instantly on send, then transition to `sent` вҶ’ `delivered` вҶ’ `read`. Feels like WhatsApp. | Have to handle the failure path: if Kafka publish fails, message goes to `failed` status with a retry button. |
| **Trust the OS / no custom expert cache (frontend bundle)** | We let the browser HTTP cache + nginx do the work. No service worker for v1. | No offline-first message browsing yet. v2 would add IndexedDB for last-N-messages-per-chat. |
| **Blue/green for production, rolling for staging** | Production traffic switch is instantaneous via Service selector patch. Smoke test green before flipping. Bad deploy is a 1-line rollback. | 2x compute cost during the switch window (~5 min). |
| **Fail-open rate limiting** | If Redis is down, allow requests through rather than hard-fail every login. Availability > strict enforcement of an abuse-prevention layer. | Brief abuse window during a Redis outage. Acceptable вҖ” we have request logs and can replay-ban after the fact. |
| **No GraphQL** | All endpoints are simple REST + WebSocket. The mobile/web client knows exactly what it needs per screen. GraphQL would add a resolver layer, an N+1 risk, and a schema-management process for negligible benefit. | Two endpoints to add when a screen needs new data. Cheap. |

### Things we explicitly chose **not** to do

- **End-to-end encryption** (Signal-style double-ratchet). v2. Adds significant complexity to group messaging and search.
- **gRPC between services**. JSON-over-HTTP and Kafka are observable enough at this scale; gRPC's binary frames make debugging harder for the team's current skill mix.
- **Service mesh (Istio/Linkerd)**. The observability we need is covered by Prometheus + structured logs. A mesh would be premature.
- **Leader election in chat-service**. There is no leader вҖ” all instances are equivalent, with Redis as the coordination point.

## 5. Quick Start

### Local (Docker Compose)

```bash
docker compose up -d
# Frontend: http://localhost:5173
# API:      http://localhost:3000
# Grafana:  http://localhost:3001 (admin/admin)
```

### Local (workspaces, no docker)

```bash
npm install
# Start the 6 dev servers (use .claude/launch.json or run individually)
npm run dev -w @chat/auth-service &
npm run dev -w @chat/chat-service &
npm run dev -w @chat/message-service &
npm run dev -w @chat/notification-service &
npm run dev -w @chat/api-gateway &
npm run dev -w frontend
```

### Tests

```bash
# Unit + integration (per service)
npm test -w @chat/auth-service
npm test -w @chat/chat-service

# Load tests (requires k6)
k6 run tests/load/auth-api.js
k6 run tests/load/websocket-chat.js
```

### Deploy

```bash
# Staging вҖ” push to main, GitHub Actions does the rest
git push origin main

# Production вҖ” tag a release
git tag v1.0.0 && git push --tags
# вҶ’ blue/green deploy with smoke tests + automatic traffic switch
```

## 6. Observability

- **Metrics**: Every service exposes `/metrics` (prom-client). Prometheus scrapes via Kubernetes pod annotations.
- **Dashboards**: Grafana provisions `Chat System Overview` (request rate, error rate, p95 latency by route, active WS, Kafka throughput, Mongo ops/s).
- **Alerts**: `ServiceDown`, `HighErrorRate (>5% over 5m)`, `HighLatency (p95 > 1s)`, `KafkaConsumerLag (> 10k)`.
- **Logs**: Structured JSON via morgan + custom logger; ship to whatever sink (Loki/ELK/CloudWatch).

## 7. Security Posture

- HTTPS/WSS only at the edge (cert-manager + Let's Encrypt in `ingress.yaml`).
- JWT signed with rotating secret; refresh tokens hashed before storage; rotation on every refresh.
- Helmet on every Express app (CSP, HSTS, no-sniff, frame-deny).
- CORS allowlist (no `*`) with credentials.
- Per-route rate limiting (auth: 10/15min, api: 60/min) with per-user keys when authenticated.
- Bcrypt cost 12 for password hashing.
- Trivy scans the source tree on every CI run (CRITICAL/HIGH).
- No secrets in images вҖ” Kubernetes Secrets + sealed-secrets recommended for production.
- S3 presigned URLs scoped to single-file PUT, 1-hour expiry.
- File type + size validation server-side before issuing the URL.
